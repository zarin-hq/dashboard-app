Extraction & Mapping Process

Systematic process for reverse-engineering Visit Salt Lake's SimpleView CMS — scraping the public site, scraping the CRM admin, and mapping how the two connect. The output is a complete blueprint for building the replacement.

The Problem

SimpleView has two systems that feed one website. We need to understand both independently, then map the connections between them.

Extraction Process

Three parallel tracks that converge into a unified data model. Steps 1-3 can run simultaneously.

Crawl the public site — every page, every widget

Hit all 1,275 pages. For each page, extract the DOM structure and identify which widgets are used, what content they contain, and what data attributes they carry (GUIDs, feed sources, layout config). Group pages by template pattern — most pages reuse the same 5-6 widget combinations.

Output: data/pages/{slug}.json — one file per page with widget tree, content, images, links, and data attributes. Plus data/templates.json — a map of unique page template patterns and which pages use each one.
AI Agent — DOM parser + classifier

Scrape the CRM admin — listings, events, maps

Navigate the SimpleView admin panel and catalog everything the CRM manages. For each entity type (events, listings, members), capture: field names, data types, relationships, filters, sort options, and any workflow states (pending, published, archived). Record network requests to reveal the internal API patterns.

Output: data/crm/events-schema.json, data/crm/listings-schema.json, data/crm/members-schema.json — data models for each CRM entity. Plus data/crm/api-patterns.json — captured XHR/fetch requests showing how the CMS talks to the CRM.
Human — admin panel access required AI Agent — network capture analysis

Map the connections — where does CRM data appear on the public site?

This is the key step. Cross-reference the public site crawl (Step 1) with the CRM schema (Step 2). For every widget that displays dynamic data (events-marquee, upcoming-events, listing cards, maps), trace where that data comes from. Identify the feed source, any filtering/sorting, and the display template.

Output: data/connections.json — a mapping of every CRM → CMS connection point. Example: {"widget": "events-marquee", "source": "crm/events", "filter": "featured=true", "display": "marquee-slide", "pages": ["/", "/events/"]}
AI Agent — cross-referencing crawl data + CRM schema

Catalog back-office workflows

Screen-record 3-5 back-office users doing their daily tasks. Feed the recordings to an AI agent that extracts: every screen visited, every action taken, every form filled, every button clicked. Build a workflow map of how staff actually use the system — not just what buttons exist, but which ones matter.

Output: data/workflows/ — one file per user role with annotated workflow maps. Identifies which CMS/CRM features are actually used daily vs. which are dead weight.
Human — schedule interviews AI Agent — video analysis + workflow extraction

Generate the replacement blueprint

Take all outputs from Steps 1-4 and produce the final artifact: a complete specification for the replacement CMS. This includes the data model (Prisma schema), the widget component library spec, the CRM entity definitions, the connection map, and the page templates. This is what we build from.

Output: blueprint/schema.prisma — database schema. blueprint/widgets/ — component specs per widget type. blueprint/templates/ — page template definitions. blueprint/api/ — API route specs for CRM data.
AI Agent — synthesis + schema generation

Proof of Concept: report.html

We took one page — /things-to-do/ — and ran the full extraction process on it to prove the approach works. Here's what it demonstrates.

We can reverse-engineer the page builder from the DOM alone

Without any access to SimpleView's source code, we extracted the complete widget tree for /things-to-do/: 19 widgets across sections, containers, and nested groups. Each widget's type, configuration, content, images, and links were captured — including internal data attributes like data-guid, data-slides-across, and data-layout that reveal how SimpleView configures each component.

What this means: We don't need their code. The DOM tells us everything about how pages are structured and configured. This scales to all 1,275 pages.

We can recreate their admin UI from scraped data

The left panel in report.html is a working page builder tree — sections, containers, and widgets with proper nesting. Click any widget and you get an editor panel with the real fields SimpleView uses: template selection, feed sources (Slides vs Feed), layout options (3-across vs 4-across), image browsers, and content tables. All populated with actual Visit Salt Lake content.

What this means: We already know what the replacement admin interface needs to look like. The editor fields we show are derived from the DOM — they represent the actual configuration options SimpleView provides. This is the spec for our page builder.

We can identify the CRM connection points

The widget tree reveals which components pull from CMS collections (slides, FAQs) vs. CRM feeds (events, listings). For example, the "Related Content" widget has data-slides-to-show="4" and its feed source is "PR - Blog Feed" — that's a CMS-managed blog. But the events-marquee and upcoming-events widgets on the homepage pull from the CRM. These connection points are visible in the DOM.

What this means: By crawling every page, we can automatically identify every place the CRM feeds data into the public site — which is exactly Step 3 of our extraction process.

We can render a live preview from pure data

The right panel is a pixel-accurate preview of /things-to-do/ built entirely from the extracted JSON — hero images, card grids, side-by-side layouts, marquees, FAQ accordions, and footer. Real images from their Cloudinary CDN. Real copy. Real links. All generated from the same data that the editor panel reads.

What this means: The scraped data is complete enough to render the page. If we can render it, we can rebuild it. The same JSON that powers the preview becomes the seed data for the replacement CMS.

Bottom line: report.html proves that for one page, we can go from "zero source code access" to "fully reverse-engineered page builder with working preview" using only DOM scraping. The extraction process in this document is how we scale that to the entire site.
